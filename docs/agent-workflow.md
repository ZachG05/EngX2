# EngX v2 — AI Agent Workflow Guide

This document defines how AI coding agents (GitHub Copilot, Claude, GPT-4, etc.) should operate within this repository.

## Start Here

Before touching any code, read these docs in order:
1. `docs/architecture.md` — directory structure, route map, where things live
2. `docs/current-priority.md` — what to work on and in what order
3. `docs/development-rules.md` — non-negotiable coding rules
4. `docs/database-schema.md` — table definitions (if the task touches the DB)

Then confirm the baseline is passing:
```bash
npm run build
```
Fix any pre-existing errors before starting your task.

## Understanding the Codebase Quickly

| Question | Where to look |
|---|---|
| What routes exist? | `src/app/` directory tree, or the Route Map in `docs/architecture.md` |
| What data shapes exist? | `src/types/index.ts` |
| What does the DB look like? | `src/lib/db/schema.ts` + `docs/database-schema.md` |
| What mock data is used? | `src/lib/problems/mock-problems.ts` |
| What Zod schemas exist? | `src/lib/validation/schemas.ts` |
| What UI primitives are available? | `src/components/ui/` (shadcn components) |

## Making Changes

### Adding a New Feature
Follow `docs/feature-template.md` step by step.

### Modifying Existing Code
- Never rename or move files without updating all imports.
- Never change `src/lib/db/schema.ts` without generating a migration (`npm run db:generate`).
- Never add `"use client"` to a file that doesn't need it — check if the component needs state/effects first.
- Never edit files in `src/components/ui/` — these are auto-generated by shadcn.

### Adding a New Route
1. Create `src/app/{route}/page.tsx` as a Server Component.
2. Keep the page thin — fetch data server-side, pass to feature components.
3. Add the route to the Route Map table in `docs/architecture.md`.

### Adding a New Component
1. Create `src/components/{feature}/ComponentName.tsx`.
2. Export it from `src/components/{feature}/index.ts`.
3. Type all props explicitly — no implicit `any`.

### Touching the Database
1. Edit `src/lib/db/schema.ts`.
2. Run `npm run db:generate` to create the migration file.
3. Update `docs/database-schema.md` to reflect the change.

### Adding Dependencies
- Run `npm install <package>` — never manually edit `package.json` dependencies.
- Check for known vulnerabilities with the GitHub Advisory Database before adding.

## Verification Checklist

Before declaring a task complete, you MUST:

- [ ] `npm run build` passes with zero TypeScript or build errors
- [ ] No `any` types introduced (check with `npm run lint`)
- [ ] All new files follow naming conventions (see `docs/development-rules.md`)
- [ ] No secrets or credentials committed
- [ ] `docs/architecture.md` route map updated if a new route was added
- [ ] `docs/database-schema.md` updated if the schema changed

## What NOT to Do

- ❌ Do NOT delete existing files unless explicitly instructed.
- ❌ Do NOT modify files in `src/components/ui/` (auto-generated by shadcn).
- ❌ Do NOT commit `.env.local` or any file with real credentials.
- ❌ Do NOT add `console.log` statements to production code.
- ❌ Do NOT use `@ts-ignore` or `any` without a comment explaining why.
- ❌ Do NOT skip `npm run build` verification.
- ❌ Do NOT put DB queries or business logic in page or component files.

## Git Commits

Format: `type: short description`

Valid types: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`

Example: `feat: add step submission with numeric tolerance checking`

Always include the Co-authored-by trailer:
```
Co-authored-by: Copilot <223556219+Copilot@users.noreply.github.com>
```
