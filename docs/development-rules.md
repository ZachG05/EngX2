# EngX v2 — Development Rules

These are non-negotiable rules for all contributors and AI agents working on this repo.

## TypeScript

- **Strict mode is ON.** No `any`, no `ts-ignore` without a comment explaining why.
- Define shared types in `src/types/index.ts`. Co-locate private types with their feature if they are never shared.
- Prefer `interface` for object shapes (e.g. `Problem`, `ProblemStep`), `type` for unions/aliases (e.g. `Difficulty`, `StepType`).
- Use `z.infer<typeof schema>` to derive types from Zod schemas rather than duplicating them.

## File & Naming Conventions

- **Components**: PascalCase file and export name (`ProblemCard.tsx`, `export function ProblemCard`)
- **Pages**: always `page.tsx` (Next.js App Router convention)
- **Utilities/Helpers**: camelCase (`mock-problems.ts`, `utils.ts`)
- **Directories**: kebab-case if multi-word (`mock-problems/`, `use-client/`)
- One component per file.
- Each feature component directory must have an `index.ts` barrel that re-exports its public components.

## Components

- Pages (`app/**/page.tsx`) must be thin: only layout structure and data fetching. No business logic.
- Server components are the default. Add `"use client"` only when you need state, effects, refs, or browser APIs.
- All component props must be typed with an explicit `interface` or `type` — no implicit `any`.
- Never import `db` or run Drizzle queries inside a Client Component or in `components/`.

## Database

- All schema is in `src/lib/db/schema.ts` (Drizzle + PostgreSQL via Supabase).
- After any schema change: run `npm run db:generate` to create a migration, then `npm run db:migrate`.
- Never modify the database directly without a matching Drizzle migration file.
- Use the `profiles` table for user metadata — never extend the Supabase `auth.users` table.
- Import the db client as `import { db } from "@/lib/db"`.

## Validation

- All Zod schemas live in `src/lib/validation/schemas.ts`.
- Every Server Action and API route must validate its input against the relevant Zod schema before touching the database.
- Do not duplicate types — use `z.infer<>` to derive TypeScript types from schemas.

## Styling

- Tailwind utility classes only. No inline `style={{}}` props.
- Use `cn()` from `@/lib/utils` for conditional class merging (not string concatenation or template literals).
- Follow the existing dark-mode design token system defined in `src/app/globals.css`.
- Never edit files in `src/components/ui/` — they are auto-generated by shadcn.

## Git

- Commit message format: `type: short description` (e.g. `feat: add step submission with tolerance checking`)
- Valid types: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`
- Never commit `.env.local` or any file with real credentials.
- Never delete an existing file without an explicit instruction to do so.

## Testing (future)

- Unit tests go in `__tests__/` co-located with the module under test, or in a top-level `tests/` directory.
- All utility functions in `src/lib/` must have unit tests when the testing infrastructure is added.
- Component tests use React Testing Library. E2E tests use Playwright.
